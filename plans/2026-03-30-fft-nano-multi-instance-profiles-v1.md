# FFT Nano Multi-Instance Profiles

## Objective

Implement a Hermes-style multi-instance profile system for FFT Nano, enabling multiple fully isolated agent instances to run from a single codebase installation. Each profile gets its own data directories (SQLite DB, groups, state, sessions, .env, logs, pi homes) while sharing the installed codebase (src, dist, skills, config, scripts).

## Research Foundation

This plan is based on a comprehensive audit of the FFT Nano codebase:

- **33 files** reference paths that would need profile awareness
- **21 scattered `process.cwd()` calls** beyond `config.ts` bypass the config layer
- The codebase conflates **code root** (skills, templates, dist) with **data root** (store, data, groups) into a single `PROJECT_ROOT = process.cwd()`
- The existing `core`/`farm` profile system (`src/profile.ts`) is a feature-flag toggle, not data isolation
- A prototype multi-instance plist (`launchd/com.nanoclaw.plist`) already exists, proving the concept
- `.env` is NOT auto-loaded at boot -- it's a persistent config store read on-demand
- The singleton lock at `data/fft_nano.lock` is per-data-dir, so separate data roots get separate locks automatically

## Phase 0: Prerequisite Cleanup (Foundation Work)

These tasks must be completed before any profile-specific code is written. They eliminate the architectural debt that makes profiles hard.

- [ ] **0.1 Separate code root from data root in `src/config.ts`**
  - Introduce `CODE_ROOT` (stays as `process.cwd()`) and `DATA_ROOT` (defaults to `process.cwd()`, overridable via `FFT_NANO_DATA_ROOT` env var)
  - `STORE_DIR`, `GROUPS_DIR`, `DATA_DIR` derive from `DATA_ROOT` instead of `PROJECT_ROOT`
  - `FFT_NANO_WEB_STATIC_DIR`, template paths, skill paths derive from `CODE_ROOT`
  - `PROJECT_ROOT` is renamed to `CODE_ROOT` to make intent explicit
  - This is the single most important architectural change -- everything else cascades from it
  - Rationale: Hermes profiles work because 119 files reference `get_hermes_home()`. FFT Nano has ~20 files referencing `DATA_DIR`/`GROUPS_DIR`/`STORE_DIR`. Making these derive from a swappable root is the same trick.

- [ ] **0.2 Eliminate fragile relative path traversals in `src/index.ts`**
  - Replace `path.join(DATA_DIR, '..', 'groups')` at `src/index.ts:622` and `src/index.ts:776-777` with direct `GROUPS_DIR` references
  - These `..` traversals assume data and groups are siblings under the same root -- they will break if data root changes
  - Rationale: These are latent bugs that will surface the moment `DATA_ROOT` diverges from `CODE_ROOT`

- [ ] **0.3 Audit and centralize all 21 scattered `process.cwd()` calls**
  - Create a shared `getCodeRoot()` function in `config.ts` for code-relative paths
  - Create a shared `getDataRoot()` function in `config.ts` for data-relative paths
  - Replace each `process.cwd()` call with the appropriate function based on whether the path is code-relative or data-relative
  - Key files to update: `pi-runner.ts:329,442`, `runtime-config.ts:112`, `index.ts:1366,1389,2373,3710,3720,3777`, `parity-config.ts:189`, `profile.ts:79`, `workspace-bootstrap.ts:187`, `tui/start.ts:14`, `tui/client.ts:311`, `onboard-cli.ts:801`, `profile-cli.ts:122`, `coding-orchestrator.ts:462`, `pi-skills.ts:605,625`, `pi-executable.ts:14`
  - Classification: code-relative (templates, skills, scripts, config files) vs data-relative (.env, secrets, state)
  - Rationale: Without this, changing `DATA_ROOT` has no effect because half the codebase bypasses it

- [ ] **0.4 Make `getDefaultDotEnvPath()` data-root-aware**
  - Change `src/runtime-config.ts:112-114` from `path.join(projectRoot, '.env')` to `path.join(getDataRoot(), '.env')`
  - Update all callers (4-5 sites) that pass explicit `projectRoot` to use data root instead
  - Rationale: Each profile needs its own `.env` with its own API keys and bot tokens

- [ ] **0.5 Add `FFT_NANO_DATA_ROOT` to shell script env flow**
  - `scripts/start.sh:67-73` and `scripts/run-launchd.sh:20-25` already source `.env` before Node starts
  - Add `FFT_NANO_DATA_ROOT` passthrough to both scripts
  - The launchd plist generated by `service.sh` should include `FFT_NANO_DATA_ROOT` in environment
  - Rationale: Shell-level env is how all config reaches Node (no dotenv auto-load)

## Phase 1: Profile Infrastructure (Core Module)

- [ ] **1.1 Create `src/profiles.ts` -- the profile CRUD engine (~400-600 lines)**
  - Profile directory structure: `<DATA_ROOT>/profiles/<name>/` containing `store/`, `data/`, `groups/`, `logs/`, `.env`
  - Default profile: when no profile is active, paths resolve to `<DATA_ROOT>/` directly (zero impact for existing users)
  - Profile resolution order: CLI `-p` flag > `active_profile` file in data root > default (no profile)
  - Validation: profile names must match `GROUP_FOLDER_PATTERN` (alphanumeric, hyphens, underscores, 1-64 chars)
  - Functions: `createProfile()`, `deleteProfile()`, `listProfiles()`, `showProfile()`, `renameProfile()`, `getActiveProfileName()`, `resolveProfileDataRoot()`, `exportProfile()`, `importProfile()`
  - Clone levels: blank (empty scaffold), `--clone` (copy .env and config only), `--clone-all` (full data copy including sessions, groups, pi homes)
  - Rationale: This is the equivalent of Hermes's `hermes_cli/profiles.py` (~900 lines). FFT Nano needs fewer lines because it has fewer surface area (no CLI wrapper aliases, no tab completion, no gateway PID management -- the service manager handles that).

- [ ] **1.2 Add `-p` / `--profile` flag parsing to `bin/fft.js`**
  - Parse `-p <name>` before any command dispatch
  - Set `FFT_NANO_DATA_ROOT` in `process.env` to `<DATA_ROOT>/profiles/<name>/` before spawning child processes
  - Add `fft profile` subcommand routing to a new `scripts/profile.sh` or direct npm script invocation
  - Collision detection: reject profile names that match existing `fft` subcommands (`start`, `dev`, `service`, `doctor`, `profile`, `onboard`, `web`, `tui`)
  - Rationale: The Hermes trick -- set the env var before any Node code runs. `bin/fft.js` is the right place because it's the CLI entry point and controls the env passed to child processes.

- [ ] **1.3 Add `fft profile` subcommand with full CRUD**
  - `fft profile status` -- show current active profile, data root, list all profiles
  - `fft profile list` -- list all profiles with creation date, size, last active
  - `fft profile create <name> [--clone|--clone-all]` -- create profile with optional cloning
  - `fft profile use <name>` -- set sticky default by writing `active_profile` file
  - `fft profile delete <name> [-y]` -- delete profile directory, warn if gateway is running
  - `fft profile show <name>` -- display profile details (paths, config summary, group count)
  - `fft profile rename <old> <new>` -- rename profile directory
  - `fft profile export <name>` -- tar.gz archive of profile directory
  - `fft profile import <archive>` -- import from tar.gz with name collision check
  - Rationale: Mirrors Hermes's `hermes profile` command surface. Each subcommand maps to a function in `src/profiles.ts`.

- [ ] **1.4 Add profile awareness to `src/doctor.ts`**
  - Add "Profile Health" section showing all profiles with config/.env/alias status
  - Warn on orphan profile directories (profile dir exists but no reference)
  - Show which profile is currently active and its data root path
  - Rationale: Hermes's `hermes doctor` includes profile diagnostics. FFT Nano's `fft doctor` should too.

## Phase 2: Isolation Guarantees (Safety)

These tasks prevent profiles from interfering with each other.

- [ ] **2.1 Per-profile singleton lock**
  - The lock file at `data/fft_nano.lock` already derives from `DATA_DIR` (`src/app.ts:466-468`)
  - Since `DATA_DIR` now derives from `DATA_ROOT` (Phase 0), and each profile gets its own `DATA_ROOT`, each profile automatically gets its own lock
  - Verify: two profiles with different data roots can run simultaneously without lock conflicts
  - Rationale: This should work automatically after Phase 0, but needs explicit verification and testing.

- [ ] **2.2 Telegram bot token conflict detection**
  - Before starting the Telegram polling loop in `src/telegram.ts:1041`, scan all other profile directories for active gateways using the same bot token
  - Implementation: check for lock files in `<DATA_ROOT>/profiles/*/data/fft_nano.lock`, read the PID, check if alive, then read that profile's `.env` for `TELEGRAM_BOT_TOKEN`
  - If a conflict is detected, log a clear error and refuse to start polling (Telegram itself will reject with "Conflict: terminated by other getUpdates request" but catching it early is better)
  - Alternative: use a shared token registry file at `<DATA_ROOT>/.token-locks.json` that maps bot tokens to profile names and PIDs
  - Rationale: This is the most dangerous conflict scenario. Two profiles polling the same bot token will cause message loss and API errors. Hermes calls these "token locks."

- [ ] **2.3 Port conflict detection**
  - Before binding TUI websocket (port 28989) and Web Control Center (port 28990), probe the port with a TCP connect attempt
  - If the port is in use by another process, log a clear error suggesting port override via `FFT_NANO_TUI_PORT` / `FFT_NANO_WEB_PORT`
  - Optionally: auto-assign port offsets based on profile index (e.g., profile "coder" gets 28989, profile "assistant" gets 28991)
  - Rationale: Two profiles binding the same port will crash the second one with `EADDRINUSE`. Hermes does "socket probe before binding."

- [ ] **2.4 WhatsApp auth state isolation**
  - WhatsApp auth state lives at `store/auth/` (derived from `STORE_DIR`)
  - Since `STORE_DIR` derives from `DATA_ROOT`, each profile automatically gets its own WhatsApp auth
  - Verify: `src/app.ts:177` uses `deps.constants.storeDir` which traces back to config
  - Verify: `src/whatsapp-auth.ts:21-23` uses hardcoded relative path `./store/auth` -- this needs to be updated to use the data root
  - Rationale: Each profile should have its own WhatsApp session. Sharing auth state between profiles would cause session conflicts.

- [ ] **2.5 Per-profile service label for launchd/systemd**
  - Derive the launchd label from the profile name: `com.fft_nano.<profile>` (default profile stays `com.fft_nano`)
  - Derive the systemd unit name: `fft-nano-<profile>.service` (default stays `fft-nano`)
  - Pass the label via `FFT_NANO_LAUNCHD_LABEL` and `FFT_NANO_SERVICE_NAME` env vars (already supported by `scripts/service.sh`)
  - Log file paths: `logs/fft_nano-<profile>.log` and `logs/fft_nano-<profile>.error.log`
  - Rationale: Each profile needs its own managed service so it can be started/stopped/restarted independently.

## Phase 3: Profile-Aware Service Management

- [ ] **3.1 Update `scripts/service.sh` for multi-profile**
  - `mac_install()`: derive plist label from `FFT_NANO_LAUNCHD_LABEL` (already supported) and set `WorkingDirectory` to `CODE_ROOT` (not profile data root -- the shell script handles env vars)
  - `linux_install()`: derive unit name from `FFT_NANO_SERVICE_NAME` (already supported)
  - Both: inject `FFT_NANO_DATA_ROOT` into the service environment so the Node process knows where profile data lives
  - Both: derive log paths from profile name to avoid log file collisions
  - Rationale: The service manager already supports configurable labels and names via env vars. The main addition is `FFT_NANO_DATA_ROOT` passthrough.

- [ ] **3.2 Update `scripts/run-launchd.sh` for profile env**
  - Source `.env` from the profile's data root (`$FFT_NANO_DATA_ROOT/.env`) instead of only from `$PROJECT_ROOT/.env`
  - Fall back to `$PROJECT_ROOT/.env` if profile `.env` doesn't exist (backward compat)
  - Pass `FFT_NANO_DATA_ROOT` through to the Node process
  - Rationale: The launchd entry point is where env vars are set before Node starts. Profile-specific `.env` must be loaded here.

- [ ] **3.3 Update `scripts/start.sh` for profile env**
  - Same `.env` sourcing logic as `run-launchd.sh`: profile `.env` first, fall back to project `.env`
  - Pass `FFT_NANO_DATA_ROOT` through to Node
  - Rationale: The foreground/dev start path must also support profiles.

- [ ] **3.4 Add `fft service` profile-aware subcommands**
  - `fft service install -p <name>` -- install service for a specific profile
  - `fft service start -p <name>` -- start a specific profile's service
  - `fft service stop -p <name>` -- stop a specific profile's service
  - `fft service restart -p <name>` -- restart a specific profile's service
  - `fft service status -p <name>` -- show status of a specific profile's service
  - `fft service list` -- show all installed profile services and their statuses
  - Rationale: Users need to manage multiple services. The `-p` flag routes to the correct service label.

## Phase 5: Workspace and Memory Isolation

- [ ] **5.1 Decide on `MAIN_WORKSPACE_DIR` per-profile strategy**
  - Option A (recommended): Each profile gets `~/nano-<profile>` as its main workspace (e.g., `~/nano-coder`, `~/nano-assistant`)
  - Option B: Each profile's workspace lives under its data root: `<DATA_ROOT>/profiles/<name>/workspace/main`
  - Option C: Share `~/nano` across profiles (not recommended -- defeats isolation)
  - Implement via `FFT_NANO_MAIN_WORKSPACE_DIR` env var, set per-profile in `.env`
  - Rationale: The main workspace contains SOUL.md, MEMORY.md, TODOS.md -- the agent's identity and context. Sharing these across profiles would cause cross-contamination.

- [ ] **5.2 Verify per-group isolation works under profiles**
  - Each profile has its own `GROUPS_DIR`, so group folders are naturally isolated
  - Each profile has its own `DATA_DIR/pi/<group>/.pi/`, so pi sessions are isolated
  - Each profile has its own `STORE_DIR/messages.db`, so chat history is isolated
  - Each profile has its own `groups/<group>/MEMORY.md`, so memory is isolated
  - Write integration test: create two profiles, register same chat JID in both, verify no data leakage
  - Rationale: These should all work automatically after Phase 0 (data root separation), but explicit verification is critical.

- [ ] **5.3 Update `scripts/backup-state.sh` and `scripts/restore-state.sh` for profiles**
  - `fft backup-state -p <name>` -- backup a specific profile's data
  - `fft restore-state -p <name>` -- restore to a specific profile
  - Default: backup/restore the active profile
  - Rationale: Backup/restore must be profile-aware or they'll overwrite the wrong data.

- [ ] **5.4 Update farm scripts for profile awareness**
  - `scripts/farm-bootstrap.sh`, `farm-validate.sh`, `farm-doctor.sh`, `farm-demo.sh`, `farm-onboarding.sh` all reference `$ROOT_DIR/data/`
  - Update to use `$FFT_NANO_DATA_ROOT/data/` when set, fall back to `$ROOT_DIR/data/`
  - Rationale: Farm state and profile data are under `DATA_DIR`, which is now per-profile.

## Potential Risks and Mitigations

1. **Phase 0 breaks existing paths (highest risk)**
   Mitigation: Phase 0 must be a separate PR with comprehensive test coverage. The default behavior (no `FFT_NANO_DATA_ROOT` set) must produce byte-identical path resolution to the current code. Run the full test suite and manual smoke test after Phase 0 before proceeding.

2. **Scattered `process.cwd()` calls missed during audit**
   Mitigation: Use `grep -rn 'process.cwd()' src/` as a CI check. Any new `process.cwd()` usage in data-path context should be flagged in code review. Add a lint rule or pre-commit hook if feasible.

3. **WhatsApp auth script (`src/whatsapp-auth.ts:21`) uses relative path**
   Mitigation: This is a known issue (research finding). Update to use data root before profiles ship.

4. **Shell scripts have inconsistent `$ROOT_DIR` usage**
   Mitigation: Introduce a `$FFT_NANO_DATA_ROOT` convention in all shell scripts with fallback to `$ROOT_DIR`. Document the convention in CLAUDE.md.

5. **Launchd plist template vs dynamic generation divergence**
   Mitigation: The dynamic plist generated by `service.sh` is the canonical path. The template plists in `launchd/` should be updated to match or removed.

6. **Port allocation strategy causes user confusion**
   Mitigation: Auto-assign port offsets with clear logging. Document the convention. Allow explicit override via env vars.

7. **Profile deletion leaves orphan data in external locations (e.g., `~/nano-coder`)**
   Mitigation: `fft profile delete` should warn about external workspace directories and offer to remove them. Never auto-delete without confirmation.

## Alternative Approaches

1. **Docker Compose per profile**: Instead of data root separation, run each profile as a separate Docker container with volume mounts. This provides stronger isolation but requires Docker for all profiles (not just agent runs) and adds operational complexity. Trade-off: stronger isolation vs simpler architecture.

2. **Symlink-based profiles**: Instead of moving data directories, create symlinks from the standard paths (`store/`, `data/`, `groups/`) to profile-specific locations. This requires no code changes but is fragile (symlinks break on cross-device moves, confuse some tools, and are invisible to casual inspection). Trade-off: zero code changes vs fragility.

3. **Environment-only isolation (no directory changes)**: Keep all data in the same directories but namespace by profile (e.g., `data/pi/<profile>/<group>/.pi/` instead of `data/pi/<group>/.pi/`). This requires more granular changes to each subsystem but avoids the two-root problem. Trade-off: more targeted changes vs no architectural refactor.

4. **Fork-per-profile**: Each profile is a separate git worktree with its own data directories. This is the most isolated approach but wastes disk space (full codebase copy per profile) and makes code updates harder. Trade-off: maximum isolation vs disk waste and update complexity.
